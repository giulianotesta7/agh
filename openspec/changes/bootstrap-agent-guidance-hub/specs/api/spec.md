# API Specification

## Purpose

HTTP API surface, resource identifiers, authentication middleware, health, and pull-manifest contract for AGH.

## Requirements

### Requirement: API version prefix and default port

All product HTTP endpoints MUST be served under `/api/v1`. The server default listen port MUST be **8912**.

#### Scenario: Versioned route namespace

- GIVEN the server is running
- WHEN a client calls `GET /api/v1/health`
- THEN the response is served by the AGH API

#### Scenario: Default port 8912

- GIVEN default server configuration without port override
- WHEN the process binds for HTTP
- THEN it listens on port 8912

### Requirement: Prefixed resource identifiers

API-exposed resource IDs MUST use stable prefixed forms (e.g., `usr_`, `prj_`, `pack_`, `asn_`) generated by the system. Clients MUST treat IDs as opaque strings.

#### Scenario: Created project returns prefixed id

- GIVEN an authenticated admin creates a project
- WHEN the API returns the project
- THEN the `id` field starts with `prj_`

### Requirement: Bearer token authentication middleware

Except for explicitly public endpoints (e.g., health), `/api/v1` routes MUST require `Authorization: Bearer <api_token>`. Invalid or missing tokens MUST yield `401 Unauthorized`. Valid tokens for deactivated users MUST yield `401` or `403` per consistent policy documented in design.

#### Scenario: Missing bearer rejected

- GIVEN a protected endpoint
- WHEN a request has no `Authorization` header
- THEN the response status is 401

#### Scenario: Valid bearer accepted

- GIVEN a user with an active token
- WHEN a request includes `Authorization: Bearer <valid_token>`
- THEN the request is attributed to that user

### Requirement: Current user endpoint

The system MUST expose `GET /api/v1/me` returning the authenticated user's email, global role, and stable user id.

#### Scenario: Me reflects caller

- GIVEN a valid bearer token for `dev@example.com` with role `member`
- WHEN `GET /api/v1/me` is called
- THEN the response includes email `dev@example.com` and role `member`

### Requirement: Health endpoint

The system MUST expose an unauthenticated health endpoint under `/api/v1` suitable for container orchestration probes.

#### Scenario: Health without auth

- GIVEN the server is running
- WHEN `GET /api/v1/health` is called without authentication
- THEN the response indicates healthy availability

### Requirement: Aggregate pull-manifest endpoint

The system MUST expose an authenticated endpoint (e.g., `GET /api/v1/projects/{project_id}/pull-manifest`) that returns all pack assignments and resolved pack contents metadata needed for a single CLI `agh pull` operation, including pack ids, resolved versions, file paths within packs, and ordering sufficient for deterministic application.

#### Scenario: Pull-manifest lists assigned packs

- GIVEN project P has packs assigned including one at `latest`
- WHEN an authorized project member requests the pull-manifest for P
- THEN the response lists each assigned pack with resolved concrete version and artifact descriptors

#### Scenario: Latest resolved to concrete version

- GIVEN assignment references `acme/onboarding@latest` and published versions include `1.0.0` and `1.1.0`
- WHEN the pull-manifest is built
- THEN the manifest includes the highest applicable SemVer (e.g., `1.1.0`), not the literal string `latest`

### Requirement: Role enforcement at API layer

Endpoints that mutate users, projects, packs, or assignments MUST enforce global and project-scoped authorization per the auth and projects specs.

#### Scenario: Member cannot publish pack

- GIVEN a user with global role `member` and no admin privileges
- WHEN they call pack publish
- THEN the response is forbidden

### Requirement: Out of scope API MVP

The API MUST NOT expose web UI routes, marketplace catalog, webhooks, or SSO/OAuth flows in this change.

#### Scenario: No marketplace route

- GIVEN MVP server
- WHEN a client requests a public pack marketplace listing
- THEN no such route exists
